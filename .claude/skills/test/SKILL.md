# 测试开发助手

基于 `bun:test` 框架的测试生成和执行指导。

## 强制输出结构

每次测试任务必须按以下顺序输出：

### 1. 测试对象清点

列出要测试的模块/函数/类：
- 名称、文件路径、职责
- 公开 API 列表

### 2. 分支清点

对每个测试对象，列出所有分支：
- 正常路径
- 边界条件
- 错误路径
- 异步场景（如有）

### 3. 测试矩阵

| 测试对象 | 场景 | 输入 | 预期输出 | 类型 |
|----------|------|------|----------|------|
| ... | 正常 | ... | ... | unit |
| ... | 边界 | ... | ... | unit |
| ... | 错误 | ... | ... | unit |

### 4. 代码输出

生成测试代码到对应位置。

### 5. 运行命令

```bash
bun test <文件路径>
```

### 6. 运行并验证

执行测试，确认全部通过。

### 7. 自查清单

- [ ] 每个公开 API 都有测试
- [ ] 正常 + 异常双向验证
- [ ] 无硬编码测试数据（使用工厂函数）
- [ ] 测试相互独立，无顺序依赖
- [ ] 异步操作正确 await
- [ ] mock/spy 在 afterEach 中清理

## 核心原则

### 有效性
- 每个测试必须验证业务规则，不写"有用"但不测业务逻辑的测试
- 正确的测试：验证计算结果、状态转换、权限检查、数据一致性
- 无用的测试：只验证函数存在、只检查返回类型

### 正反双测
- 正向：输入合法数据 → 验证正确结果
- 反向：输入非法数据 → 验证正确拒绝

### 真实操作
- 通过公开 API 触发操作，不直接修改内部状态
- 使用工厂函数创建测试数据（`createTest*`、`make*`）

## 测试文件组织

```
packages/<pkg>/
  src/
    foo.ts
  __tests__/
    foo.test.ts
```

## 禁止行为

1. 不写只检查"不抛错"的测试
2. 不写只验证类型的测试（TypeScript 编译器已做）
3. 不 mock 被测对象本身
4. 不在测试间共享可变状态
5. 不写依赖执行顺序的测试
6. 不跳过失败的测试（`test.skip`）除非有 TODO 注释说明原因
7. 不写超过 50 行的单个测试用例（拆分）
