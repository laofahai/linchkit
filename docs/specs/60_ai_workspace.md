## 1. Motivation

LinchKit is an AI-Native Software Capability Runtime. The meta-model (Entity, Action, Rule, State, Event, EventHandler, View, Flow, Relation) is inherently machine-readable. Yet today, AI coding tools (Claude Code, Codex, Cursor, Trae, etc.) must manually parse source files to understand a LinchKit project.

The goal: **any AI coding tool that opens a LinchKit project should immediately understand the domain model, available operations, and project conventions — and be able to operate the running system through standardized protocols.**

Traditional scaffolding is not the priority. AI Workspace is. The system should teach AI tools about itself, both at development time and at runtime.

## 2. Two-Layer Architecture

AI interacts with LinchKit through two distinct layers with a strict boundary between them.

```
┌──────────────────────────────────────────────────────────────┐
│                    Dev-time AI (Layer 1)                      │
│  AI helps you write code                                     │
│  Channel: File system + AGENTS.md + MCP dev tools            │
│  Tools: Claude Code, Cursor, Codex, Trae                     │
│  Quality gate: Human reviews code via PR                     │
├──────────────────────────────────────────────────────────────┤
│                    Runtime AI (Layer 2)                       │
│  AI interacts with running system                            │
│  Channel: MCP transport, GraphQL, REST API                   │
│  Tools: AI agents, chatbots, automation                      │
│  Quality gate: CommandLayer + ProposalEngine                 │
└──────────────────────────────────────────────────────────────┘
```

### 2.1 Clear Boundary

| What AI touches | Layer | Governance | Example |
|-----------------|-------|------------|---------|
| **Definitions** (code) | Dev-time | PR review | `defineEntity()`, `defineAction()` |
| **Data** (records) | Runtime | CommandLayer (auth → permission → ...) | Create purchase request, approve order |
| **Structure** (schema changes) | Runtime → Dev-time | Proposal → Approval → Code/Overlay | Add field to entity, new rule |

**Principle: "AI Never Modifies Production Directly."**

- Dev-time AI writes `defineXxx()` code → human reviews via PR → deploy
- Runtime AI executes Actions → CommandLayer enforces auth/permission
- Runtime AI wanting structural changes → creates Proposal → human approves → either Runtime Overlay (small, Spec 59) or code change (large, PR)

### 2.2 Information Flow

```
                    Dev-time                          Runtime
                    ────────                          ───────
Source of truth:    Source code (.ts files)            Database + Runtime Overlay
AI reads via:       AGENTS.md, MCP dev server,        MCP runtime, GraphQL, REST
                    file system
AI writes via:      Code generation → PR              Actions → CommandLayer
                                                      Proposals → Approval
Ontology source:    Static analysis of defineXxx()    OntologyRegistry (live)
```

## 3. Dev-time AI Features

### 3.1 AGENTS.md Auto-Generation

Every LinchKit project should have an AGENTS.md that teaches AI tools the full project context. This file is auto-generated by `linch agents-md` and updated on each `linch dev` startup.

**Content structure:**

```markdown
# <Project Name>

## Framework
This is a LinchKit project (AI-Native Software Capability Runtime).

## Meta-Model
- Entity: Data structure definitions (defineEntity)
- Action: Sole write entry point (defineAction), named verb_noun
- Rule: Declarative conditions + effects (defineRule)
- State: Finite state machine per entity (defineState)
- Event: Domain events (defineEvent)
- EventHandler: Reactions to events (defineEventHandler)
- View: UI rendering config (defineView)
- Flow: Multi-step durable workflows (defineFlow)
- Relation: Bidirectional entity relationships (defineRelation)

## Conventions
- Entity naming: snake_case
- Action naming: verb_noun (e.g. submit_request, approve_order)
- All mutations go through Actions — never write to DB directly
- System fields (auto-managed): id, tenant_id, created_at, updated_at,
  created_by, updated_by, _version

## Installed Capabilities
### <capability_name> (v<version>)
- Entities: <list with field summaries>
- Actions: <list with input/output summaries>
- Rules: <list with condition summaries>
- States: <state machine overview>
- Relations: <relationship descriptions>
- Views: <view types>

## Available MCP Dev Tools
<tool list with descriptions>

## Common Patterns
<project-specific patterns extracted from capabilities>

## Anti-Patterns
<things to avoid, extracted from rules and conventions>
```

**Generation sources:**

| Section | Source |
|---------|--------|
| Framework / Meta-Model | Static template (ships with `@linchkit/cli`) |
| Conventions | `linchkit.config.ts` + static template |
| Installed Capabilities | Scan `capabilities/` and `addons/` directories, parse `defineCapability()` exports |
| MCP Dev Tools | MCP dev server tool registry |
| Common Patterns | Extracted from methodology files in `.linchkit/methodology/` |
| Anti-Patterns | Extracted from rules + static template |

**CLI commands:**

```bash
linch agents-md              # Generate/regenerate AGENTS.md
linch agents-md --watch      # Regenerate on file changes (used by linch dev)
```

### 3.2 MCP Development Server

A development-time MCP server distinct from `cap-adapter-mcp` (which is runtime). This server helps AI coding tools understand and work with the project during development.

**Startup:** Automatically started by `linch dev` or manually via `linch mcp-dev`.

**Architecture:**

```
linch dev
  ├── Elysia server (:3001)           ← Runtime (cap-adapter-server)
  ├── Vite dev server (:3000)         ← UI (cap-adapter-ui)
  └── MCP dev server (stdio/SSE)      ← Dev-time AI tools (this spec)
        ├── Uses OntologyRegistry internally
        ├── Has access to source files
        └── Can run CLI commands (test, lint, typecheck)
```

#### 3.2.1 Tools

**Discovery tools** (read-only, help AI understand the project):

| Tool | Description | Returns |
|------|-------------|---------|
| `linchkit_list_entities` | List all entities with field count, action count, relation count | `EntitySummary[]` |
| `linchkit_describe_entity` | Full entity description: fields, actions, rules, states, relations, views | `EntityDescriptor` (from OntologyRegistry) |
| `linchkit_list_actions` | List all actions with entity, input params, description | `ActionSummary[]` |
| `linchkit_describe_action` | Full action definition with input/output schema, rules, side effects | `ActionDescriptor` |
| `linchkit_list_relations` | List all relations with cardinality and connected entities | `RelationSummary[]` |
| `linchkit_explain_relation` | Explain relationship graph for an entity (all incoming/outgoing) | `RelationGraph` |
| `linchkit_describe_capability` | Full capability structure: entities, actions, rules, views, config | `CapabilityDescriptor` |
| `linchkit_list_capabilities` | List all installed capabilities with summaries | `CapabilitySummary[]` |

**Validation tools** (check without applying):

| Tool | Description | Returns |
|------|-------------|---------|
| `linchkit_validate_definition` | Validate a proposed entity/action/rule definition (TypeScript source) without applying it | `ValidationResult` with errors/warnings |
| `linchkit_check_quality` | Run biome lint + typecheck, return results | `QualityReport` |
| `linchkit_run_tests` | Run tests, optionally scoped to a capability | `TestReport` |

**Generation tools** (produce code):

| Tool | Description | Returns |
|------|-------------|---------|
| `linchkit_generate_entity` | Generate `defineEntity()` code from natural language description | TypeScript source string |
| `linchkit_generate_action` | Generate `defineAction()` code from description | TypeScript source string |
| `linchkit_generate_capability` | Generate full capability scaffold from description | File tree with source strings |

> Generation tools produce code strings that the AI tool writes to disk. They never write files directly.

#### 3.2.2 Resources

| URI | Description | MIME |
|-----|-------------|------|
| `linchkit://ontology` | Full ontology as JSON (all entities, actions, rules, relations) | `application/json` |
| `linchkit://entity/{name}` | Single entity definition as JSON | `application/json` |
| `linchkit://capability/{name}` | Capability definition as JSON | `application/json` |
| `linchkit://conventions` | Project conventions and patterns | `text/markdown` |

#### 3.2.3 Prompts

| Name | Description |
|------|-------------|
| `design_entity` | Guided entity design workflow (asks clarifying questions) |
| `design_capability` | Guided capability design workflow |
| `diagnose_error` | Analyze an error message in LinchKit context |

#### 3.2.4 Implementation

```typescript
// packages/cli/src/mcp-dev-server.ts
import { OntologyRegistry } from '@linchkit/core'
import { Server } from '@modelcontextprotocol/sdk/server'

export function createMcpDevServer(options: {
  ontology: OntologyRegistry
  projectRoot: string
  capabilities: CapabilityDefinition[]
}): Server {
  const server = new Server({
    name: 'linchkit-dev',
    version: '1.0.0',
  })

  // Register tools, resources, prompts
  registerDiscoveryTools(server, options.ontology)
  registerValidationTools(server, options.projectRoot)
  registerGenerationTools(server, options.ontology)
  registerResources(server, options.ontology)
  registerPrompts(server)

  return server
}
```

**Transport:** stdio by default (for Claude Code, Cursor). SSE endpoint optionally exposed on `linch dev --mcp-port 3002`.

**Configuration in `.mcp.json`:**

```json
{
  "mcpServers": {
    "linchkit-dev": {
      "command": "bun",
      "args": ["./node_modules/.bin/linch", "mcp-dev"],
      "env": {
        "DATABASE_URL": "..."
      }
    }
  }
}
```

### 3.3 Project Introspection CLI

CLI commands that both humans and AI tools can use to understand the project.

| Command | Description | Output |
|---------|-------------|--------|
| `linch describe` | Project overview: entity count, action count, capability list, relation graph summary | Formatted text |
| `linch describe entity <name>` | Detailed entity info: fields, actions, rules, states, relations | Formatted text |
| `linch describe action <name>` | Action details: input/output, rules, side effects | Formatted text |
| `linch describe capability <name>` | Capability structure | Formatted text |
| `linch describe relations` | Relation graph overview | Formatted text or DOT graph |
| `linch agents-md` | Regenerate AGENTS.md from current project state | Writes file |
| `linch mcp-dev` | Start MCP development server (stdio) | MCP server process |
| `linch mcp-dev --sse --port 3002` | Start MCP dev server with SSE transport | HTTP server |

**Implementation:** All `linch describe` commands use OntologyRegistry internally. They load capabilities from the project, build the ontology, and format output.

### 3.4 AI-Friendly Error Messages

When Actions fail, Rules reject, or validation errors occur, error messages must include enough context for an AI agent to understand and fix the issue autonomously.

**Current error format (Spec 33):**

```typescript
{
  code: 'VALIDATION_ERROR',
  message: 'Validation failed',
  details: { field: 'amount', message: 'Must be positive' }
}
```

**Enhanced format for AI consumption:**

```typescript
interface AIFriendlyError {
  // Existing fields (Spec 33)
  code: string
  message: string
  httpStatus: number

  // AI context (new)
  context: {
    entity?: string           // Which entity was involved
    action?: string           // Which action was attempted
    field?: string            // Which field caused the issue
    constraint?: string       // Which constraint failed (rule name, validation name)
    expected?: string         // What was expected
    actual?: string           // What was provided
    suggestion?: string       // What to change (human-readable)
    relatedDocs?: string[]    // Relevant spec/doc references
  }
}
```

**Examples:**

```typescript
// Rule rejection
{
  code: 'RULE_REJECTED',
  message: 'Rule "budget_check" rejected action "submit_request"',
  context: {
    entity: 'purchase_request',
    action: 'submit_request',
    constraint: 'budget_check',
    expected: 'amount <= department.remaining_budget (50000)',
    actual: 'amount = 75000',
    suggestion: 'Reduce amount to 50000 or less, or request budget increase for department',
  }
}

// State transition failure
{
  code: 'INVALID_STATE_TRANSITION',
  message: 'Cannot transition from "draft" to "approved"',
  context: {
    entity: 'purchase_request',
    field: '_state',
    expected: 'Valid transitions from "draft": ["submitted"]',
    actual: 'Attempted transition to "approved"',
    suggestion: 'Submit the request first (action: submit_request), then approve',
  }
}
```

Error context is populated by the engine that produces the error (ActionEngine, RuleEngine, StateMachine). The `context` field is always present in development mode. In production, it is included only when the caller is authenticated as an AI agent (determined by CommandLayer auth slot).

### 4.2 AI Agent Standard Workflow

Define the standard interaction pattern for an AI agent operating on a running LinchKit system.

```
┌─────────────────────────────────────────────────────────┐
│                 AI Agent Workflow                         │
│                                                          │
│  1. DISCOVER                                             │
│     list_entities → understand domain model              │
│     describe_entity → deep-dive specific entities        │
│     list_relations → understand entity graph             │
│                                                          │
│  2. QUERY                                                │
│     GraphQL queries → read data                          │
│     get_execution_log → understand history               │
│                                                          │
│  3. ACT                                                  │
│     execute_action → write data (Actions only)           │
│     All writes go through CommandLayer pipeline           │
│                                                          │
│  4. OBSERVE                                              │
│     get_execution_log → see results                      │
│     get_recent_executions → monitor patterns             │
│                                                          │
│  5. PROPOSE (structural changes only)                    │
│     create_proposal → suggest schema/rule changes        │
│     Human approval required                              │
│                                                          │
│  6. LEARN (Spec 55 integration)                          │
│     get_insights → access AI analysis results            │
│     get_anomalies → see detected anomalies               │
└─────────────────────────────────────────────────────────┘
```

**Key constraint:** Steps 1-2 (read) and 3-4 (write+observe) go through CommandLayer. Step 5 (propose) always requires human approval. Step 6 requires Spec 55 Evolution System.

### 4.3 AI Agent Identity

AI agents are identified in the system via the standard auth mechanism (Spec 10/10a). Each AI agent:

- Has an `Actor` with `type: 'ai_agent'`
- Is assigned roles and permissions like any other actor
- Has its actions logged with `created_by` / `updated_by` referencing the agent actor
- Can be rate-limited per the CommandLayer rate-limit middleware

**AI agent registration** is handled by `cap-adapter-mcp` using the MCP Client Registry (Spec 58).

## 6. Security Model

### 6.1 Dev-time Security

- MCP dev server runs locally (stdio transport by default)
- No authentication required for local stdio — the developer's machine is the trust boundary
- SSE transport (`--sse`) should bind to localhost only by default
- Generation tools produce code strings; they never write to disk or execute code
- All generated code goes through normal PR review before deployment

### 6.2 Runtime Security

- All runtime AI interactions go through CommandLayer (7-slot pipeline)
- AI agents authenticate via standard mechanisms (JWT, API key)
- Per-agent permission scoping via roles (Spec 10)
- Per-client tool visibility via MCP Client Registry (Spec 58)
- Structural change proposals always require human approval (ProposalEngine)
- AI agent actions are fully audited in execution logs

### 6.3 Separation of Concerns

| Concern | Dev-time | Runtime |
|---------|----------|---------|
| Trust model | Developer's machine | Server-side auth |
| Write target | Source code (files) | Database (records) |
| Review gate | PR review | CommandLayer + ProposalEngine |
| Rollback | `git revert` | Action compensation / Proposal rollback |
| Blast radius | Next deployment | Immediate (data) |

## 7. Integration with Existing Specs

### 7.1 Spec 43 — OntologyRegistry

The MCP dev server and CLI introspection commands are thin wrappers around OntologyRegistry. They do not duplicate ontology logic.

```
OntologyRegistry.describe(name)  →  linchkit_describe_entity tool
OntologyRegistry.listEntities()  →  linchkit_list_entities tool
OntologyRegistry.toJSON()        →  linchkit://ontology resource
```

### 7.2 Spec 55 — Evolution System

The AI agent standard workflow (Section 4.2, Step 6: Learn) integrates with the Evolution System's Insight and Awareness layers. AI agents can:

- Query insights generated by the Awareness layer
- Access anomaly detection results
- Use insights to inform their own actions

### 7.3 Spec 15 — AI Developer Experience

This spec supersedes and refines the dev-time portions of Spec 15. Specifically:

| Spec 15 concept | Spec 60 refinement |
|-----------------|-------------------|
| CLAUDE.md auto-generation | AGENTS.md auto-generation (tool-agnostic, follows agents.md standard) |
| MCP tools (runtime) | Split into dev-time MCP server (this spec) + runtime MCP (cap-adapter-mcp) |
| Skills | Retained as-is in Spec 15; MCP dev server prompts complement them |
| Prompt management | Retained as-is in Spec 15 |

### 7.4 Spec 57 — Addon Architecture

AGENTS.md generation scans addons (OCA-grouped capabilities) to document their entities, actions, and relations. Capabilities with `autoInstall: true` are noted in the generated AGENTS.md.

### 7.5 Spec 58 — MCP Client Registry

Runtime AI agent management (registration, per-client authorization, tool visibility) is handled by the MCP Client Registry. This spec does not redefine that — it references it for runtime agent identity.

## 9. What This Spec Does NOT Cover

- **Traditional scaffolding** (`linch init` project templates) — explicitly deprioritized
- **AG-UI protocol** (Agent ↔ UI real-time) — separate spec, handled by `cap-adapter-ag-ui`
- **A2A protocol** (Agent ↔ Agent) — separate spec, handled by `cap-adapter-a2a`
- **AI model selection / provider management** — covered by Spec 36
- **Evolution System engines** (Sense, Memory, Awareness, Insight) — covered by Spec 55
- **Runtime Overlay** mechanism — assumed as a dependency, not re-specified here

## 10. Success Criteria

1. An AI coding tool opening a LinchKit project with a generated AGENTS.md can correctly explain the domain model without reading source files
2. The MCP dev server provides enough context for an AI tool to generate a valid `defineEntity()` on the first attempt in >80% of cases
3. `linch describe` output is sufficient for a developer (or AI) to understand the project structure in under 30 seconds
4. Runtime AI agents can complete the full Discover → Query → Act → Observe workflow without human intervention (within their permission scope)
5. Structural change proposals from AI agents always go through human approval — zero bypass paths
