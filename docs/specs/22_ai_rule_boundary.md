# AI 与 Rule Engine 的边界设计规范

## 1. 核心区分

```
Rule Engine：确定性裁决 — 我知道规则是什么，给我数据，我来判断
AI：概率性推理 — 我理解意图、发现模式、提出建议
```

| | Rule Engine | AI |
|--|------------|-----|
| 确定性 | 100%，同输入同输出 | 概率性，可能不一致 |
| 速度 | 毫秒级 | 秒级 |
| 成本 | 零（代码执行） | 每次调用有成本（API 费用） |
| 运行时机 | 每次 Action 都跑 | 按需或定期 |
| 适合 | 已知的、明确的业务规则 | 模糊的、需要理解的、创造性的 |

## 2. Rule Engine 负责什么

**运行时，每次 Action 执行都跑，零成本：**

- 金额超过阈值需要审批 → 明确的条件判断
- 状态迁移合法性检查 → 状态机规则
- 必填字段校验 → 确定性校验
- 重复提交拦截 → 幂等检查
- 权限校验 → 角色/权限组判断
- 数据一致性约束 → 跨字段/跨记录校验
- 自动数据补充 → enrich effect
- 触发后续动作 → execute_action effect

## 3. AI 负责什么

**非实时，建议性质，需要 Proposal 流程确认：**

- 模式发现 → "这个部门最近采购频率异常高"
- 规则建议 → "建议增加规则：连续3次被驳回的供应商自动标记"
- 阈值优化 → "根据历史数据，建议把审批阈值从1万调到5万"
- 系统设计 → 从需求描述生成 Schema / Action / Rule
- 意图理解 → 自然语言转 Action 调用
- 异常检测 → 分析 Execution Log 发现异常模式
- 流程优化 → 分析瓶颈，建议改进

## 4. 边界原则

> **凡是能写成确定性条件的，用 Rule Engine。凡是需要理解、推理、创造的，用 AI。**

> **AI 的输出如果要影响生产系统，必须固化成 Rule 或其他定义，通过 Proposal 流程发布。**

```
AI 发现模式
    ↓
AI 提出建议（Proposal）
    ↓
Validation（验证建议是否合理）
    ↓
人工审批
    ↓
固化为 Rule / Schema / Action 定义
    ↓
Rule Engine 在运行时确定性执行
```

## 5. AI 不应该做什么

- ❌ 在运行时做业务判断（太慢、不确定、有成本）
- ❌ 直接修改生产数据
- ❌ 替代 Rule Engine 做已知规则的判断
- ❌ 每次 Action 执行都调用 AI

## 6. AI 可以做什么（通过正确的路径）

- ✅ 通过 MCP 调用 Action（作为用户使用系统）
- ✅ 通过 GraphQL 查询数据（理解系统状态）
- ✅ 生成 Proposal（建议系统变更）
- ✅ 定期分析 Execution Log（发现问题和优化点）
- ✅ 辅助开发者编写 Capability 代码

## 7. 进化循环

```
Observe（AI 观察运行数据）
    ↓
Analyze（AI 分析模式、发现问题）
    ↓
Propose（AI 生成 Proposal）
    ↓
Validate → Approve → Release（治理流程）
    ↓
Run（Rule Engine 确定性执行新规则）
    ↓
Observe（继续观察...）
```

这就是 LinchKit 的"进化循环"。AI 负责智慧（发现、建议），Rule Engine 负责执行（确定、高效）。两者通过 Proposal 流程衔接。
