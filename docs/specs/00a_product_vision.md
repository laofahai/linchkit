# 产品愿景 — 用户说，要有光

> Tracking milestones:
> - long-range product vision reference across multiple milestones
>
> Related issues:
> - No dedicated open issue is currently tracked for this spec.
>
> Execution source of truth: GitHub milestones and issues.

> **"用户说，要有光。"**
>
> 系统不是被设计出来的，是从使用中长出来的。

## 1. 核心理念

传统软件开发的逻辑是：**先设计，再开发，最后交付给用户使用。**

LinchKit 的逻辑是反过来的：**用户先说需求，系统立刻出现。**

```
传统：需求分析 → 系统设计 → 编码 → 测试 → 部署 → 使用
LinchKit：说 → 有
```

这不是"低代码"——低代码仍然要求用户理解表、字段、流程图等技术概念。LinchKit 追求的是：**用户只需要描述业务，技术细节完全由 AI 翻译为系统构件。**

中间产物——Schema、Action、Rule、Flow、Link、View——对用户透明。用户看到的是业务在运行，而不是一堆配置。

## 2. 生长模型

系统的演进不是版本迭代，而是**有机生长**。

### 2.1 播种：多种起点

系统不止一种起点。根据用户的情况，播种方式不同：

**方式 A：一句话播种**

```
用户："我要管采购"
  ↓ AI 翻译
  ↓ defineEntity('purchase_request', { ... })
  ↓ defineAction('create_purchase_request', { ... })
  ↓ 自动生成 View、GraphQL、REST
  ↓
系统出现了。
```

适合：明确知道要做什么，但规模很小的场景。

**方式 B：行业冷启动包**

```
用户："我们是制造企业，大概200人"
  ↓ 匹配行业包：manufacturing-starter
  ↓ 安装：采购、库存、生产工单、质检 四个 Capability
  ↓ 预置：常用 Schema + Rule + Flow + View
  ↓
一套基础系统出现了。
```

大部分用户不会真的从零开始。冷启动包提供了**80%的通用起点**，用户在此基础上裁剪和生长——删掉不需要的、调整不合适的、补充缺少的。

冷启动包分两个层次：

**职能包（横向，跨行业通用）：**

采购、库存、人事、财务、项目管理、客户管理、审批流……这些环节的核心流程不因行业而异。职能包封装的是**通用业务逻辑**。

**行业包（纵向，职能包的组合 + 行业定制）：**

行业包不是从零写的，而是职能包的组合，叠加行业特定的 Rule/Flow 覆写。

```
职能包（通用）         行业包（组合 + 定制覆写）
──────────────       ──────────────────────
📦 采购               🏭 制造业 = 采购 + 库存 + 生产工单 + 质检
📦 库存                              （采购包叠加：来料质检 Rule）
📦 人事               🏥 医疗 = 采购 + 人事 + 患者管理
📦 财务                              （采购包叠加：合规审批 Flow）
📦 项目管理           🛒 零售 = 采购 + 库存 + 会员 + 促销
📦 客户管理                          （库存包叠加：多门店调拨 Flow）
📦 审批流
```

这意味着：一个制造企业装了"制造业包"，其中的采购模块和一个零售企业装的采购模块**底层是同一个职能包**，只是上面叠加的行业规则不同。

冷启动包本质是 **Capability 的预组合**，不是独立的产品。它遵循同样的增量原则：包里的每个 Capability 都可以独立卸载，每个 Schema/Rule 都可以被覆盖。用户既可以装完整的行业包，也可以只挑几个职能包自行组合。

**方式 C：数据倒推**

```
用户导入一批 Excel / CSV
  ↓ AI 分析数据结构和内容
  ↓ 推断 Schema 定义（字段名、类型、关系）
  ↓ 生成 Proposal
  ↓ 用户确认
  ↓
Schema 从数据中长出来了。
```

适合：已经有数据但没有系统的团队。先有数据，后有模型。

**方式 D：Fork 已有 Capability**

```
用户："我需要类似采购管理，但流程不太一样"
  ↓ Fork cap-purchase → cap-my-purchase
  ↓ 在此基础上修改 Rule、Flow、View
  ↓
系统从别人的经验中长出来了。
```

适合：有参考对象的场景。不必从头描述，站在前人肩上。

**共同点：** 无论哪种起点，种子播下后的生长方式都一样。起点可以不同，但生长机制是统一的。任何时刻的系统都是完整可用的——区别只在功能范围，不在完成度。

### 2.2 生长：被痛点驱动

系统不是按照预先设计的蓝图扩展，而是**被实际使用中的痛点驱动**。每一次生长都有明确的触发原因：

| 阶段 | 用户说 | 系统长出 |
|------|--------|----------|
| Day 1 | "我要管采购" | Schema + CRUD Action + 默认 View |
| Week 1 | "老有人填错供应商" | Validation Rule |
| Week 1 | "超过5万要老板批" | Approval Flow |
| Month 1 | "采购单要和入库单关联" | Link |
| Month 1 | "每次审批完都要手动通知仓库" | Automation Rule |
| Month 2 | "想看每月采购趋势" | 聚合 View |
| Month 3 | "新来的同事不该看到金额" | Permission Rule + Data Masking |

注意：**没有一步是提前规划的。** 每一步都是用户在使用中发现"缺了什么"，然后一句话补上。

### 2.3 演进：活的系统

系统不只是被动响应用户需求——它是**活的**。

传统软件是死的：写好了就固定了，除非有人改它。LinchKit 能感知、能记忆、能理解自己、能发现机会、能安全地改变自己。它同时拥有四样传统软件不具备的东西：

- **自我认知**：OntologyRegistry 让系统知道自己的完整结构
- **全量记忆**：PersistentEventBus 记录一切发生过的事
- **推理能力**：AI 能对模式进行归因和翻译
- **安全自改**：Proposal 机制让系统能修改自己而不崩溃

基于这四个能力，系统形成一个五层的**生命循环**（详见 [Spec 55](./55_evolution_system.md)）：

```
Sense（感知）    — 多维信号持续流入（API/MCP/UI/Server/EventBus）
    ↓               传感器 Capability 化，业务模块自带行业传感器
Memory（记忆）   — 使用模式积累，自动建立基线，发现偏移
    ↓               摩擦信号（放弃、撤销、重复）优先采集
Awareness（意识）— 系统知道"什么对这个业务重要"
    ↓               不是人定义的，是从使用中涌现的
Insight（洞察）  — 有证据支撑的事实，附带完整证据链
    ↓               拉模式为主，仅高置信+高影响才推送
Proposal（进化） — AI 把 Insight 翻译成可执行的系统变更
    ↓               附带回测、影响预览、回滚条件
用户确认 → 系统进化 → 新的信号流入 → 循环继续
```

举例：

```
Week 1-4: 系统默默记录采购审批的驳回数据，建立基线（驳回率 12%）
Week 5:   驳回率升至 28%，超出基线 2σ
          系统意识到采购审批是核心流程（★★★★★），这个偏移值得关注
          生成 Insight："驳回率上升，主因金额超标(73%)"
          用户看到 Insight，点击"生成建议"
          AI 翻译为 Proposal："添加规则——金额超预算时提交前提醒"
          附带回测：过去30天23条会触发，18条确实被驳回
          用户确认 → 影子模式运行一周 → 效果验证 → 全量生效
Week 8:   驳回率降至 10%，系统记录正向反馈
```

**没有人告诉系统"你应该关注驳回率"。系统自己感知到了。**

## 3. "说"的多种形式

"用户说，要有光"中的"说"，不只是自然语言对话。用户通过**多种方式**表达需求，系统都能响应：

### 3.1 对话：直接描述

用户用自然语言说出需求，AI 解析意图，生成系统构件。

> "创建一个请假申请的流程，需要主管审批，超过3天还需要HR审批"

最直觉的方式，适用于用户已经清晰知道自己要什么的场景。

### 3.2 选择：从行业包中挑选

用户不需要从零描述，而是从行业冷启动包中选择接近的起点，然后裁剪。

> 浏览 Capability Hub → 找到"制造业-采购管理" → 安装 → 删掉不需要的审批层级 → 调整字段

这是"说"的一种——通过**选择和裁剪**来表达需求。

### 3.3 导入：用数据说话

用户把已有的 Excel、CSV 导入系统，AI 从数据中推断结构。

> 导入3年的采购记录 → AI 发现：有供应商、金额、部门、审批人四列
> → 建议 Schema + Link 定义

数据本身就是一种表达——它说明了"我们一直在管什么"。

### 3.4 行为：从操作中推断

系统观察用户的操作模式，推断出潜在需求。

> 用户连续3天手动把「待处理」的采购单按金额排序后逐一审批
> → 系统建议："是否需要自动按金额排序待审批列表？"

用户没有"说"，但行为本身就是一种表达。

### 3.5 涌现：从数据中发现

当系统积累了足够的运行数据，会涌现出用户自己都没意识到的模式。

> 系统发现：周五下午提交的采购单审批通过率只有23%，而周一上午是89%。
> → "建议：金额超过1万的采购申请避免在周五下午提交，历史通过率较低。"

这不是用户"说"出来的需求，是系统从数据中"长"出来的智慧。

### 3.6 总结

```
         主动程度                     "说"的形式
         ────────────────────────────────────────────
高    █  对话：直接描述需求
      █  选择：从行业包中挑选
      █  导入：用已有数据表达
      █  行为：操作模式被观察
低    █  涌现：数据自己说话
```

从上到下，用户越来越"不需要说"，系统越来越"自己知道"。这是一个渐进的过程——初期以显式表达为主，随着数据积累，隐式表达和涌现的比例逐渐增加。

## 4. 设计原则

从这个愿景出发，推导出以下设计原则：

### 4.1 零假设启动

不预设用户需要什么。`linch init --bare` 创建的是一个空壳——没有任何业务逻辑，但已经是一个可运行的系统。一切从用户的第一句话开始。

### 4.2 增量永远可行

在任何时刻，都可以无破坏地添加新能力。这意味着：
- 新 Schema 不影响已有 Schema
- 新 Rule 不破坏已有流程
- 新 Capability 安装即用，卸载无残留
- 弱依赖优先：缺少某个 Capability 时功能优雅降级，而不是报错

### 4.3 AI 翻译，而非 AI 替代

AI 的角色是**翻译官**——把用户的业务语言翻译成系统构件。AI 不做业务决策，不替用户判断该不该批准采购单。

```
用户的世界          AI 翻译层           系统的世界
─────────────    ──────────────    ─────────────
"管采购"     →    defineEntity()   →    Schema
"超5万要批"  →    defineRule()     →    Rule
"通知仓库"   →    defineFlow()     →    Flow
"两个表关联" →    defineRelation()     →    Link
```

用户始终活在业务语言中，不需要跨过技术鸿沟。

### 4.4 进化即代码

进化的产出不是数据库记录，而是 **TypeScript 文件**。

系统进化出的每一条 Rule、每一个 Schema 扩展、每一个 Flow，最终都会"毕业"为 `defineXxx()` TS 代码，提交到 Git。这意味着：

- **可追溯**：Git 历史记录了系统是如何一步步进化的
- **可 review**：每次进化都可以被代码审查
- **可复现**：克隆仓库就能得到完整的进化历史，不依赖任何平台
- **可复用**：进化出的规则可以被提取为 Capability，分享给其他项目
- **AI 友好**：TS 代码是结构化数据，AI 天然理解，无需自然语言中转

**系统的"基因"属于用户，不属于平台。** 这是和封闭式 SaaS 平台的根本区别。

### 4.5 结构化一切

LinchKit 选择 `defineXxx()` TypeScript 而非 YAML/JSON/自然语言，因为**结构化数据对 AI 最友好**。

整条进化链路都是结构化的——从原始事件（有 schema）、传感器信号（有类型定义）、Insight（有证据链接口）到 Proposal（有 defineXxx 代码），AI 在每一层都能精准操作，不需要猜测。

这也是 OntologyRegistry 的价值所在：它让 AI 不只能"使用"系统，还能"理解"系统——知道系统有哪些 Schema、哪些 Action、它们之间的关系是什么。系统对 AI 是**完全透明的**。

### 4.6 Proposal 保护安全

每一次系统变更——无论来自用户对话还是 AI 建议——都通过 **Proposal** 机制：

1. AI 生成变更方案（新的 Schema/Rule/Flow...）
2. 以 Proposal 形式呈现给用户，包含变更预览和影响分析
3. 用户确认后才生效

这保证了"说了就有"的流畅体验，同时避免了 AI 幻觉或误解造成的破坏。

### 4.7 每个状态都是完整的

系统在任何时刻都不存在"未完成"的状态。Day 1 只有一个 Schema 的系统和 Month 6 有几十个 Schema 的系统，都是完整可用的。区别只在于功能范围，而不在于完成度。

这意味着没有"上线"的概念——系统从第一句话开始就在运行。

## 5. 元模型

### 5.1 完整元模型

LinchKit 的元模型分两层——**骨架**（系统运行的结构）和**生命系统**（系统进化的能力）：

```
骨架（现有）：
  Schema + Action + Rule + State + Event + EventHandler + View + Flow + Link

生命系统（新增，详见 Spec 55）：
  Sense      — 多维感知能力（defineSensor），Capability 可扩展
  Memory     — 使用记忆（基线、偏好、摩擦记录、业务节奏、进化历史）
  Awareness  — 自我意识（重要性图谱 + 行业知识 defineKnowledge）
  Insight    — 洞察（有证据链的事实，晋升规则过滤噪音）
  Proposal   — 进化提案（Insight 的可执行翻译，毕业为 TS 代码）
```

骨架是系统**是什么**，生命系统是系统**怎么长**。

### 5.2 扩展点

生命系统的每个层都通过 Capability 的 `extensions` 机制扩展，和骨架的扩展方式一致：

```typescript
defineCapability({
  extensions: {
    // 骨架扩展（现有）
    fieldTypes: [...],
    viewTypes: [...],
    services: [...],

    // 生命系统扩展（新增）
    sensors: [defineSensor(...)],             // 注册传感器
    sensorExtensions: [extendSensor(...)],    // 扩展已有传感器
    knowledge: defineKnowledge(...),           // 注册行业知识
  },
})
```

这意味着：
- 装一个业务 Capability → 系统多了业务逻辑 **+ 对应的感知能力**
- 装一个行业包 → 系统多了行业模块 **+ 行业传感器 + 行业知识库**
- 系统的"身体"和"感官"同步成长

## 6. 与技术架构的映射

这个愿景如何落地到 LinchKit 的技术架构：

| 愿景 | 技术实现 |
|------|----------|
| "说"→"有" | AI Intent Parser → Proposal → defineXxx() 生成 |
| 职能冷启动包 | 跨行业通用 Capability（采购、人事、财务…） |
| 行业冷启动包 | 职能包组合 + 行业定制覆写，Capability Hub 一键安装 |
| 数据倒推 | 数据导入 → AI Schema 推断 → Proposal |
| 零假设启动 | `--bare` 模式，零 Capability 可运行 |
| 增量无破坏 | Capability 弱依赖 + graceful degradation |
| 活的系统 | 五层生命循环：Sense → Memory → Awareness → Insight → Proposal（Spec 55）|
| 多维感知 | SignalBus + 传感器 Capability 化，业务模块自带行业传感器 |
| 摩擦优先 | 摩擦信号（放弃/撤销/重复）优先于成功日志 |
| 进化即代码 | Proposal 验证通过 → 毕业为 defineXxx() TS 文件 → Git commit |
| 结构化一切 | 全链路结构化：Event → SensorSignal → Insight → Proposal → TS 代码 |
| 行业知识库 | defineKnowledge() — 行业先验知识，加速 Awareness，提升 Proposal 质量 |
| Skill 复用 | 进化系统通过 MCP/Tool 调用外部能力（搜索、代码分析、知识库） |
| Proposal 保护 | Proposal → Validate → Shadow → Promote（Spec 02/55）|
| 每个状态完整 | 三层 Source of Truth（Design/Deploy/Runtime）|
| AI 翻译层 | OntologyRegistry + MCP Tools + builder Agent |
| Fork 生长 | Capability Hub 浏览 + fork + 本地定制 |

## 7. 两种生长

LinchKit 的生长有两种驱动力，共存互补：

| | 显式生长 | 隐式进化 |
|--|---------|---------|
| 驱动力 | 用户 | 系统自身 |
| 触发 | 用户说出需求 | 系统感知到模式 |
| AI 角色 | 翻译（业务语言→系统构件） | 翻译（Insight→Proposal） |
| 决策者 | 用户确认 | 用户确认 |
| 时间线 | 即时 | 积累后涌现 |

初期以显式生长为主（用户说→系统出现）。随着使用数据积累，隐式进化的比例逐渐增加——系统越来越"自己知道该怎么长"。

**更深层的可能：** 当显式和隐式生长持续积累，系统可能涌现出**第三种生长**——跨领域的。一个采购模块的摩擦模式，和一个库存模块的异常信号，在 Awareness 层关联后，可能发现一个**谁都没想到的业务机会**。这不是"用户说了"，也不是"某个模块自己发现的"，而是整个有机体的涌现。

## 8. 为什么它永远不会过时

传统软件过时的本质是**系统和业务脱节**——业务变了，系统没跟上。

LinchKit 不会脱节，因为它不是一个"被交付的产品"，而是一个**和业务共生的有机体**：

- **业务规模变了** → 系统感知到操作量和复杂度的变化，建议新的规则和流程
- **行业法规变了** → 安装新的行业知识包，系统立刻知道该关注什么
- **组织架构变了** → 系统观察到审批链路和角色分布的变化，建议调整权限和流程
- **AI 能力进化了** → Proposal 翻译质量变好，系统的进化速度也随之加快

它不是"一次实施就结束"的软件——它是**一直在实施中的软件**，或者说，实施这个概念本身消失了。系统永远在跟着公司走。

而且，因为所有业务逻辑都是 `defineXxx()` 结构化代码，系统的"基因"可以被 Fork、被复用、被组合。一个制造企业进化出的质检规则，可以被另一个制造企业直接使用。**每一家公司的进化，都在丰富整个生态。**

更深一层：因为系统理解自己的结构（OntologyRegistry）+ 理解使用模式（Memory）+ 拥有行业知识（Knowledge），它甚至能发现**用户自己都没想到的场景**——不是"按需求开发"，而是"需求从数据中涌现"。这是传统软件永远做不到的事。

## 9. 一句话总结

LinchKit 不是一个需要"实施"的软件。它是一个活的有机体——用户的每一句话是阳光，使用数据是土壤中的养分，系统在两者的共同滋养下，自然生长为用户需要的样子。

**用户说，要有光。然后，就有了光。而光照之处，万物生长。**
