# 文档治理规范

> Tracking milestones:
> - `M5: Platform Maturity & AI Evolution`
>
> Related issues:
> - GitHub Issue `#124` — Documentation governance
>
> Execution source of truth: GitHub milestones and issues.

## 1. 定位

`25_documentation.md` 定义的是“文档作为系统能力的产物形态”。

本文定义的是“文档如何被持续维护、校验、归档和发布”，避免出现：

- 文档重命名后引用未更新
- README 索引与实际文件不一致
- devlog 有记录但索引未同步
- 历史文档与正式文档边界不清
- 自动生成文档与人工文档责任不明

目标不是写更多文档，而是让文档体系可持续、可协作、可校验。

## 2. 文档分类

LinchKit 文档分为四类：

### 2.1 总览文档

面向项目整体理解与导航。

示例：

- `README.md`

### 2.2 正式规范

面向架构、实现和治理的当前有效规则。

示例：

- `specs/*.md`

### 2.3 开发记录

面向开发会话接续、决策追踪和上下文回溯。

示例：

- `devlog/*.md`

### 2.4 历史文档

保留设计演进过程，但不再作为当前实现依据。

示例：

- `LinchKit_架构讨论整理稿_v0.1.md`

## 3. 文档状态

每份文档应有明确状态。推荐使用以下四种：

| 状态 | 含义 | 是否作为当前依据 |
|------|------|------------------|
| `active` | 当前有效文档 | 是 |
| `draft` | 草稿，内容未冻结 | 否 |
| `deprecated` | 即将废弃，仍保留兼容信息 | 原则上否 |
| `historical` | 历史记录，仅供参考 | 否 |

### 3.1 状态标注规则

- `README.md` 默认视为 `active`
- `specs/*.md` 默认视为 `active`，除非文首明确标注 `draft` 或 `deprecated`
- `devlog/*.md` 默认视为记录型文档，不参与状态标注
- 历史稿必须在文首显式标注 `historical`

### 3.2 Draft 文档要求

如果某份 spec 尚未达成一致，应在文首增加显式标注：

```markdown
> **状态：draft**  
> 本文档仍在讨论中，未作为当前实现依据。
```

草稿允许存在待定项，但必须写明影响范围。

## 4. Source of Truth 规则

### 4.1 规范优先级

当多个文档出现冲突时，优先级如下：

1. `specs/*.md`
2. `README.md`
3. `devlog/*.md`
4. 历史文档

### 4.2 文档与代码的关系

- 设计阶段：`specs/*.md` 是架构与实现约束的 source of truth
- 实现阶段：代码和自动生成产物成为运行时事实来源
- 如果代码已实现但 spec 未更新，应视为文档缺陷，必须补齐

## 5. 索引维护规则

### 5.1 README 索引

当新增、删除、重命名正式规范时，必须同步更新：

- `README.md` 中的“规范文档索引”
- 相关里程碑中的文档链接

### 5.2 devlog 索引

每新增一条 `devlog` 记录，必须同步更新 `devlog/README.md` 中的“记录索引”。

索引规则：

- 最新记录放最上面
- 每条索引一行
- 格式统一为：

```markdown
- 2026-03-21_001_xxx.md - 一句话摘要
```

## 6. 重命名与编号调整规则

文档重命名、编号调整是最容易引入漂移的操作，必须遵循以下流程：

1. 修改目标文件名
2. 全局搜索旧文件名和旧编号
3. 更新所有正文引用
4. 更新 `README.md` 索引
5. 如影响历史说明，保留一条 devlog 记录

### 6.1 不允许的做法

- 只改文件名，不改引用
- 只改 README，不改 specs 交叉引用
- 用“读者自己理解”代替全局修复

## 7. 链接校验

### 7.1 校验范围

以下文档必须参与链接校验：

- `README.md`
- `specs/*.md`
- `devlog/README.md`

### 7.2 校验要求

- 所有相对路径链接必须存在
- 重命名后不得保留旧编号链接
- 文档索引与实际文件名必须一致

### 7.3 建议自动化

在 CI 中增加文档校验脚本，至少覆盖：

- Markdown 相对链接存在性检查
- `README.md` 中 `specs/*.md` 链接可达性检查
- `devlog/README.md` 索引项存在性检查

## 8. 自动生成文档与人工文档的责任边界

### 8.1 自动生成文档负责什么

自动生成文档负责：

- Capability 结构清单
- Schema / Action / Rule / State / View 概览
- 变更历史与版本信息
- Execution Trace 等运行痕迹

自动生成文档的目标是：

- 不漂移
- 可搜索
- 适合机器与 AI 读取

### 8.2 人工文档负责什么

人工文档负责：

- 背景
- 决策理由
- 使用方式
- 约束说明
- 风险说明

人工文档的目标是：

- 解释“为什么”
- 说明“如何判断”
- 记录“哪些地方不能想当然”

### 8.3 禁止责任错位

- 不要把结构清单长期手写在人工文档里
- 不要把设计决策只埋在自动生成产物里
- 不要用 devlog 替代正式规范

## 9. 历史文档治理

历史文档可以保留，但必须满足：

- 文首显式标注“历史文档”
- 写明当前生效文档位置
- 不从 README 主索引中作为正式规范入口暴露

如果历史稿仍保留在项目根目录，建议在文首写清楚：

```markdown
> **状态：historical**  
> 本文是讨论过程记录，正式设计以 `specs/` 为准。
```

## 10. 与里程碑的关系

### M0

- 建立文档分层
- 建立 README 与 devlog 索引规则
- 建立重命名与链接修复流程

### M1

- 链接校验纳入 CI
- 文档状态标记纳入规范
- 自动生成文档与人工文档目录结构固定

### M2

- 文档搜索与文档查询能力完善
- AI 可区分 `active / draft / historical` 文档状态
